Theory & Fundamentals

What is a Hash Table?

A Hash Table maps keys to values using a hash function that turns a key into an array index (a bucket). Done well, insert, delete, and lookup are O(1) average.

The catch is collisions: two keys hashing to the same bucket. How you resolve them (chaining vs open addressing) and your load factor decide real-world performance.

The hash function

// map key → bucket index
index = hash(key) % capacity

// good hash: fast, uniform, deterministic
// load factor α = n / capacity
// resize (rehash) when α > ~0.75
Why Hash Tables? They turn O(n) lookups into O(1): the single most common way to speed up an algorithm. Two Sum, frequency counts, dedup, caches, sets, symbol tables — all hash tables underneath.
Collision Resolution
🔗
Separate Chaining
Each bucket holds a linked list (or vector) of entries that hash there.
Use: simple, handles high load factor. std::unordered_map.
➡️
Linear Probing
On collision, step to the next slot (i+1, i+2…) until an empty one.
Use: cache-friendly, but clustering.
🔢
Quadratic / Double Hash
Probe by i², or use a second hash for the step, reducing clustering.
Use: open addressing with better spread.
Terminology
Hash Function
Maps a key to an integer index, ideally uniformly.
Bucket / Slot
An array cell where entries hashing to that index live.
Collision
Two distinct keys mapping to the same bucket.
Load Factor (α)
n / capacity. High α → more collisions → slower ops.
Rehashing
Growing the table and re-inserting all keys when α gets high.
Clustering
Consecutive filled slots in probing, hurting performance.

Advantages & Disadvantages

Advantages

  • O(1) average insert, search, delete.
  • Turns nested-loop O(n²) into single-pass O(n).
  • Flexible keys (strings, tuples with custom hashers).
  • Foundation of sets, maps, caches, dedup.

Disadvantages

  • No ordering — can't do range queries or sorted iteration.
  • Worst case O(n) with adversarial keys / bad hash.
  • Extra memory for empty slots (load factor < 1).
  • Rehash pauses; poor cache locality with chaining.

#include <bits/stdc++.h>
using namespace std;

int main() {
    unordered_map<string,int> freq;   // key→value, O(1) avg
    freq["apple"]++;
    freq["apple"]++;
    if (freq.count("apple")) cout << freq["apple"]; // 2

    unordered_set<int> seen;          // membership, O(1) avg
    seen.insert(5);
    bool has = seen.count(5);       // true

    freq.reserve(1000);              // pre-size to avoid rehash
    cout << freq.load_factor();       // current α
    return 0;
}

// Two Sum — O(n) with a hash map
vector<int> twoSum(vector<int>& a, int target){
    unordered_map<int,int> seen;   // value → index
    for (int i = 0; i < a.size(); i++) {
        int need = target - a[i];
        if (seen.count(need)) return {seen[need], i};
        seen[a[i]] = i;
    }
    return {};
}

// Chaining hash set (illustrative)
struct HashSet {
    static const int B = 769;   // prime capacity
    vector<int> bucket[B];
    int idx(int k){ return (k % B + B) % B; }
    void add(int k){ auto& c = bucket[idx(k)];
        if (find(c.begin(),c.end(),k)==c.end()) c.push_back(k); }
    bool has(int k){ auto& c = bucket[idx(k)];
        return find(c.begin(),c.end(),k)!=c.end(); }
};

Key Patterns
Hash map = O(1) lookup. The universal speedup: replace "search the array again" (O(n) inner loop) with a hash map lookup (O(1)). Two Sum, subarray-sum-equals-k, longest consecutive sequence, group anagrams all use this.
Frequency counting. map[x]++ in one pass gives you counts for anagrams, majority element, top-K, and first-unique problems. For a small fixed alphabet, an int[26]/int[128] array is even faster.

Key Takeaways
⚡O(1) average: hash(key) % capacity picks a bucket. Insert, search, delete are constant time when the hash spreads keys well.
💥Collisions are inevitable: resolve with chaining (lists per bucket) or open addressing (probing). Keep load factor below ~0.75 and rehash.
🚀The universal speedup: a hash map turns O(n²) nested loops into O(n) — Two Sum, frequency counts, dedup, prefix-sum tricks.
📐No ordering: need sorted iteration or range queries? Reach for a balanced BST (std::map) instead — O(log n) but ordered.
