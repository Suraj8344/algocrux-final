Tree Recursion — Bottom-Up Approach

Solve children first, then combine their returned values at the parent — the answer bubbles up from the leaves — O(N) time

What is bottom-up recursion?

In a bottom-up traversal each call first recurses into both children, waits for their return values, then combines them into its own answer and returns that upward. Nothing is decided until the subtrees report back.

This is the natural shape for anything defined in terms of subtree results: height (1 + max of child heights), whether a subtree is balanced, node/leaf counts, subtree sums. It is exactly a postorder walk: left, right, then act on the node.

// bottom-up: children return, parent combines
solve(node):
  if node == null: return base
  L = solve(node.left)
  R = solve(node.right)
  return combine(L, R, node.val)

Bottom-up powers Maximum Depth (104), Balanced Binary Tree (110), and Count Complete Tree Nodes (222). Same postorder skeleton, different combine step.
Open Problem
Time Complexity
O(N)

Each node returns exactly once; the combine step is O(1) per node.

Space Complexity
O(H)

Recursion stack depth equals the tree height H (O(N) if skewed).
